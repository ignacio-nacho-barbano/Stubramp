# @stubramp/api

Fastify 5 API backed by **Neon Postgres** via **Prisma 7** (with the `pg` driver adapter). Deploys to **Fly.io**.

## Stack

| Concern        | Choice                                                              |
| -------------- | ------------------------------------------------------------------- |
| Server         | Fastify 5 (ESM, `tsx` in dev, `tsc` build)                          |
| Database       | Neon serverless Postgres (cloud) — local Docker Postgres optional   |
| ORM            | Prisma 7 (query compiler + `@prisma/adapter-pg` over node-postgres) |
| Runtime driver | `pg` pool, opened once at module scope (`src/db.ts`)                |

## Quick start (Neon — default)

The Neon project is already created and linked (`.neon`, git-ignored). Connection
strings are pulled into `.env.local` by the Neon CLI:

```bash
neonctl env pull            # refresh the current branch's DATABASE_URL etc.
pnpm install                # also runs `prisma generate`
pnpm db:migrate             # apply migrations to the linked Neon branch
pnpm dev                    # http://localhost:3001  (GET /, GET /health)
```

`GET /health` runs `SELECT 1` against the DB — use it to confirm connectivity.

### Branch-first workflow

Each feature gets its own isolated Neon branch (mirrors `git`):

```bash
neonctl checkout dev-my-feature   # create/switch branch + pull its env into .env.local
```

No local DB needed — this is the recommended workflow.

## Connection model

Neon exposes two connection strings; we use each where it belongs:

- **`DATABASE_URL`** — pooled (PgBouncer). Used by the app at runtime (`src/db.ts`),
  correct for a long-running host like Fly.
- **`DATABASE_URL_UNPOOLED`** — direct. Used by Prisma migrations (`prisma.config.ts`),
  because DDL can't run reliably over the pooler.

Both `prisma.config.ts` and `src/env.ts` load `.env.local` first, then `.env`.

> Prisma 7 note: connection config lives in `prisma.config.ts`, **not** the schema.
> `directUrl` and the `adapter` config field were removed — driver-adapter migrations
> work automatically. The generated client is TypeScript at `src/generated/prisma/`
> (git-ignored, regenerated by `postinstall` / `prisma generate`).

## Local Postgres via Docker (optional, offline dev)

Not required — Neon covers cloud + branching. Use this only for fully-local/offline work.
Requires a container runtime (Docker Desktop, OrbStack, podman, …).

```bash
pnpm db:up                  # start local Postgres (docker compose, port 5432)
pnpm db:migrate:docker      # migrate the local DB
pnpm dev:docker             # run the API against the local DB
pnpm db:down                # stop  (pnpm db:reset also wipes the volume)
```

Setting `USE_LOCAL_DB=1` makes both loaders read the committed `.env.docker`
(localhost:5432, `sslmode=disable`) instead of Neon's `.env.local`. Plain
`pnpm dev` / `pnpm db:migrate` always use Neon.

## Scripts

| Script                | Purpose                                          |
| --------------------- | ------------------------------------------------ |
| `dev`                 | Watch-mode server against Neon                   |
| `build`               | `prisma generate && tsc` → `dist/`               |
| `start`               | Run the built server (`node dist/index.js`)      |
| `db:generate`         | Regenerate the Prisma client                     |
| `db:migrate`          | Create + apply a migration (dev)                 |
| `db:deploy`           | Apply migrations without prompts (CI/production) |
| `db:studio`           | Open Prisma Studio                               |
| `db:up` / `db:down`   | Start / stop local Docker Postgres               |
| `db:reset`            | Stop local Postgres **and wipe its data**        |
| `dev:docker`          | `dev` against local Docker Postgres              |
| `db:migrate:docker`   | `db:migrate` against local Docker Postgres       |

## Environment variables

See [`.env.example`](./.env.example). Secrets live in `.env.local` (git-ignored,
Neon-managed); local Docker creds live in the committed `.env.docker`.

| Variable                | Used by              | Notes                                        |
| ----------------------- | -------------------- | -------------------------------------------- |
| `DATABASE_URL`          | app runtime          | Pooled (PgBouncer) connection                |
| `DATABASE_URL_UNPOOLED` | Prisma migrations    | Direct connection                            |
| `USE_LOCAL_DB`          | env loaders          | `1` → use `.env.docker` instead of `.env.local` |
| `HOST` / `PORT`         | server               | Defaults `0.0.0.0` / `3001`                  |

## Deploying to Fly.io

Use region `iad` (Ashburn, VA) — co-located with the Neon `us-east-1` project for low latency.

```bash
fly launch --no-deploy                              # primary region: iad
fly secrets set DATABASE_URL="<neon pooled url>"    # + DATABASE_URL_UNPOOLED if migrating on deploy
fly deploy
```

Run `pnpm db:deploy` in the release step to apply migrations.
